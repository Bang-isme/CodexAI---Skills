# Anti-Generic Patterns

Use this reference when an answer sounds correct but could apply to almost any project.

## Common Failure Modes

- advice without named files, commands, or boundaries
- improvement language without a measurable change
- one recommended path with no tradeoff surface
- abstract "best practices" disconnected from repo reality
- padded summaries that hide missing evidence

## Repair Moves

1. Replace every vague verb with a concrete action.
2. Name the artifact that changes.
3. State what evidence would prove the claim.
4. State one realistic failure mode.
5. If the advice still sounds reusable for any repo, it is still too generic.

## Phrases To Distrust

- "follow best practices"
- "ensure scalability"
- "improve performance"
- "enhance maintainability"
- "robust solution"
- "seamless workflow"
- "optimize the process"

These phrases are allowed only when followed immediately by repo-specific details.

## Specificity Test

Before finalizing, ask:

- Which file or script makes this real?
- Which command verifies it?
- Which metric, count, or observable signal changes?
- What is the smallest concrete next step?
